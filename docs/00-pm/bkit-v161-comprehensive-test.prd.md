## 4. PM-Research: 테스트 방법론 조사

### 4.1 CC Plugin 테스트 Best Practice

| 방법 | 설명 | bkit 적용 |
|------|------|-----------|
| **claude -p (prompt mode)** | CLI에서 프로그래밍 방식으로 CC 호출 | E2E 테스트: 스킬 트리거, 에이전트 호출, PDCA 흐름 |
| **node 직접 실행** | require()로 모듈 로드 후 함수 단위 테스트 | Unit 테스트: 순수 함수, 파서, 계산 로직 |
| **eval.yaml + runner.js** | 스킬별 eval 정의 + 자동 실행기 | Benchmark 테스트: 28개 스킬 일괄 검증 |
| **Hook 시뮬레이션** | stdin으로 JSON 입력 → stdout 검증 | Hook 테스트: user-prompt-handler.js 동작 검증 |

### 4.2 테스트 자동화 분류

| 분류 | 자동화 가능 | 수동 필요 | 이유 |
|------|:-----------:|:---------:|------|
| **Unit** | 100% | 0% | 순수 함수, 입출력 명확 |
| **Integration** | 90% | 10% | Config 로드는 환경 의존 |
| **E2E** | 80% | 20% | claude -p 호출 가능하지만 결과 판정에 인간 검토 필요 |
| **UX** | 30% | 70% | 사용자 경험 흐름은 주관적 판단 필요 |
| **Philosophy** | 60% | 40% | 원칙 준수 여부는 정량화 어려움 |
| **Security** | 90% | 10% | disallowedTools 파싱은 자동화 가능 |
| **Regression** | 95% | 5% | 기존 TC 재실행으로 대부분 자동화 |

```
  ├── Performance tests (PRF-001~008)
  └── Regression tests (REG-001~020)
```

#### 5.10.2 claude -p 호출 전략 (비용 최적화)

| 전략 | 설명 | 적용 |
|------|------|------|
| **Node 우선** | claude -p 없이 node로 실행 가능한 테스트는 node 직접 | Unit, Integration, Performance |
| **배치 호출** | claude -p E2E 테스트를 하나의 세션에서 연속 실행 | PDCA full cycle |
| **결과 캐싱** | 동일 결과 기대 TC는 캐싱하여 재실행 방지 | Regression |
| **선별 실행** | P0만 먼저 실행, P1/P2는 P0 통과 후 | 전체 |
