### Critical Discovery: Issue #41 — CTO Lead Broken (2026-03-06)

**GitHub**: [bkit Issue #41](https://github.com/popup-studio-ai/bkit/issues/41)

CC v2.1.69에서 "Subagents cannot spawn other subagents" 제한이 하드 코딩됨. bkit의 CTO Lead(`cto-lead`)는 subagent로 spawn된 후 `Task(enterprise-expert)`, `Task(bkend-expert)` 등을 spawn하려 하므로 **nested spawn**으로 차단됨.

```
현재 (v1.6.0, 비동작):
Main Session → Agent(cto-lead) → Task(enterprise-expert)  ← BLOCKED
                 ↑ subagent         ↑ nested spawn

수정 후 (v1.6.1, Approach A):
Main Session (CTO 역할) → Agent(enterprise-expert)  ← OK (1-level only)
                          → Agent(bkend-expert)      ← OK
                          → Agent(gap-detector)      ← OK

수정 후 (v1.6.1, Approach B - Agent Teams):
Main Session (Team Lead) → Teammate(enterprise-expert)  ← OK (Teams API)
                          → Teammate(bkend-expert)      ← OK
                          → Teammate(gap-detector)      ← OK
                          → Shared Task List + Mailbox
```

**CC 공식 아키텍처 제약 (2026-03 확인)**:

| Constraint | Source | Impact |
|-----------|--------|--------|
| Subagents cannot spawn other subagents | [CC Official Docs](https://code.claude.com/docs/en/sub-agents) | CTO subagent가 팀원 spawn 불가 |
| No nested teams | [Agent Teams Docs](https://code.claude.com/docs/en/agent-teams) | Teammate도 자체 팀 생성 불가 |
| Team agent에서 `skills`, `hooks` 무시 | [CC #30703](https://github.com/anthropics/claude-code/issues/30703) | Agent Teams 마이그레이션 시 제약 |
| Custom agent definitions 부분 지원 | [CC #24316](https://github.com/anthropics/claude-code/issues/24316) | `disallowedTools`, `model` 동작, `skills`/`hooks` 미동작 |

## 2. Alternatives Explored

### 2.1 Overall Strategy

### Approach A: Surgical Fix + Framework Completion — **Selected**

P0 버그 4건(3 기존 + 1 신규 #41)을 정밀 수정하고, CTO 오케스트레이션 아키텍처를 재설계하며, Evals를 실구현.

- **Pros**: 최소 변경으로 최대 CE 점수 향상 (+10점), CC v2.1.69+ 호환성 즉시 복구, 1-2주 완료 가능
- **Cons**: Should-Have 일부 미포함 가능
- **Best for**: v1.6.1 마이너 릴리스의 품질 목표 달성 + CTO Team 복구

### Approach B: Full Rewrite of Intent/Team Subsystems — **Rejected**

- 변경 범위가 크고 (40+ 파일), 리그레션 위험 높음, 3-4주 소요
- v2.0.0 major 릴리스에 적합

### Approach C: Evals-First Approach — **Rejected**

- P0 버그가 Evals 완성까지 방치됨 (No Guessing 위반 지속)
- 의존성 체인이 불필요하게 길어짐

### 2.2 CTO Orchestration Architecture (Issue #41 해결)

#### Option 1: Main Session as CTO (A안) — **Fallback**

Main Opus 세션이 직접 CTO 역할을 수행하고, subagent를 1-level만 spawn.

```
Main Session (Opus, CTO role via pdca skill)
  ├── Agent(enterprise-expert)     ← 1-level subagent (OK)
  ├── Agent(bkend-expert)          ← 1-level subagent (OK)
  ├── Agent(gap-detector)          ← 1-level subagent (OK)
  └── Agent(code-analyzer)         ← 1-level subagent (OK)
```

- **Pros**: 즉시 동작, 코드 변경 최소, CC GA 기능만 사용
- **Cons**: 2단계 위임 체인 여전히 차단 (qa-strategist→gap-detector 등 6경로), PM Lead도 동일 문제
- **한계**: A안만으로는 불완전 → A'변형(2단계 평탄화) 필요하지만 기능 저하 수반
- **용도**: B안 사전 검증 실패 시 fallback

#### Option 2: CC Agent Teams Migration (B안) — **v1.6.1 Selected**

CC Agent Teams(TeamCreate)를 사용하여 Main Session이 Team Lead로 동작. 각 teammate가 독립 세션이므로 **2단계 위임 체인이 그대로 보존**.

```
Main Session (Team Lead)
  ├── Teammate: enterprise-expert (독립 세션) ✅
  │     └── Agent(infra-architect)    ← 1단계 of teammate ✅
  ├── Teammate: qa-strategist (독립 세션) ✅
  │     ├── Agent(gap-detector)       ← 1단계 of teammate ✅
  │     ├── Agent(code-analyzer)      ← 1단계 of teammate ✅
  │     └── Agent(qa-monitor)         ← 1단계 of teammate ✅
  ├── Teammate: frontend-architect ✅
  ├── Teammate: security-architect ✅
  │     └── Agent(code-analyzer)      ← 1단계 of teammate ✅
  └── Shared: Task List + Mailbox

PM Team (동일 패턴):
  ├── Teammate: pm-discovery ✅
  ├── Teammate: pm-strategy ✅
  ├── Teammate: pm-research ✅
  └── Teammate: pm-prd ✅
```

- **Pros**: 중첩 문제 완전 해결 (2단계 위임 평탄화 불필요), teammate 간 직접 소통, CC 공식 팀 기능
- **Cons**: Agent Teams 실험적, `memory` frontmatter 미검증, 토큰 비용 증가, 세션 재개 미지원
- **전제조건**: 사전 검증 스파이크 (memory 지원 + 2단계 위임 검증)
- **구현**: `lib/team/coordinator.js`에서 `buildAgentTeamPlan()`, pdca SKILL.md에서 TeamCreate 기반 오케스트레이션

#### Option 3: Hybrid (Subagent + Agent Teams) — **v2.0.0 (Future)**

연구/분석은 subagent로, 구현/협업은 Agent Teams로 하이브리드 운영.

- **Pros**: 각 패턴의 장점 활용, 유연한 팀 구성
- **Cons**: 복잡성 높음, 두 시스템 동시 관리
- **전제조건**: CC Agent Teams GA(General Availability) 전환

**선택 근거**: B안이 CTO팀과 PM팀의 **2단계 위임 체인을 유일하게 완전 보존**. A안은 1단계만 해결하여 6개 위임 경로가 여전히 차단됨. 사전 검증 스파이크로 memory/2단계 위임을 확인 후 B안 확정, 실패 시 A'안 fallback.

### Layer 2: Evals Implementation (Core) — 20 hours

#### M-04: Skill Evals 28/28 실구현

**2.1 Runner.js 실구현** (`evals/runner.js`)

현재 `runEval()`이 항상 `pass: true`를 반환하는 stub 상태. 실제 eval 로직 구현:

```javascript
// runner.js — runEval() 핵심 로직
async function runEval(skillName, evalName) {
  const definition = loadEvalDefinition(skillName);
  if (!definition) return { pass: false, details: { error: `No eval found for ${skillName}` } };

  // 1. YAML에서 eval 정의 파싱
  const evalDef = parseEvalYaml(definition.content);

  // 2. prompt 파일 로드
  const promptPath = path.join(path.dirname(definition.path), evalDef.evals[0].prompt);
  const prompt = fs.readFileSync(promptPath, 'utf8');

  // 3. expected 파일 로드
  const expectedPath = path.join(path.dirname(definition.path), evalDef.evals[0].expected);
  const expected = fs.readFileSync(expectedPath, 'utf8');

  // 4. 평가 기준 확인
  const criteria = evalDef.evals[0].criteria || [];

  // 5. 결과 검증 (criteria 기반 pass/fail)
  const result = evaluateAgainstCriteria(prompt, expected, criteria);

  return { pass: result.pass, details: { ...result, skill: skillName } };
}
```

### Layer 3: Config-Code Sync (Trust) — 8 hours

#### M-05: orchestrationPatterns → PHASE_PATTERN_MAP 동기화

**파일**: `lib/team/orchestrator.js`

**현재 문제**: `PHASE_PATTERN_MAP` (line 19-34)이 `bkit.config.json`의 `orchestrationPatterns`과 동일한 값을 독립적으로 정의. Config 변경 시 코드가 따라가지 않음.

**수정 방안**:
```javascript
// lib/team/orchestrator.js
// PHASE_PATTERN_MAP 상수 제거 → config에서 런타임 로드

function selectOrchestrationPattern(phase, level) {
  const { getConfig } = require('../core');

  // Config가 Single Source of Truth
  const configPattern = getConfig(`team.orchestrationPatterns.${level}.${phase}`);
  if (configPattern) return configPattern;

  // Fallback: Starter = single, default = leader
  if (level === 'Starter') return 'single';
  return 'leader';
}
```

**영향 범위**: orchestrator.js, 기존 API 시그니처 변경 없음
